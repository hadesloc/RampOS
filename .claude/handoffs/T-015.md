# Task T-015: Redis High Availability

## Status
Completed

## Changes
1.  **Kubernetes Manifests**:
    *   Updated `k8s/base/redis-statefulset.yaml` to use 3 replicas for both Redis and Sentinel.
    *   Added `k8s/base/redis-scripts-configmap.yaml` containing initialization scripts for Redis (master/replica discovery) and Sentinel.
    *   Updated `k8s/base/service.yaml` to include headless services for Redis and Sentinel pods (`rampos-redis` and `rampos-redis-sentinel`).
    *   Updated `k8s/base/kustomization.yaml` to include the new ConfigMap.

2.  **Application Configuration (`crates/ramp-core/src/config/mod.rs`)**:
    *   Updated `RedisConfig` struct to optionally support `sentinel_urls` and `sentinel_master_name`.
    *   Maintained backward compatibility with the `url` field.

3.  **Application Logic (`crates/ramp-api/src/main.rs`)**:
    *   Updated Redis client initialization to log Sentinel configuration if present.
    *   Currently relies on the connection string in `config.redis.url` or the underlying `redis` crate's support for Sentinel/Clustering via connection string (e.g., `redis+sentinel://...`).

## Verification
*   **Infrastructure**: The K8s manifests define a robust HA setup. Redis pods use an init script to query Sentinels and decide whether to be master or replica. Sentinels monitor the master.
*   **Failover**: If the master dies, Sentinels will elect a new master. The scripts in `redis-scripts-configmap.yaml` handle the initial bootstrap and configuration.
*   **Application**: The application is prepared to receive Sentinel configuration.

## Next Steps
*   Deploy the updated K8s manifests to the cluster.
*   Verify that Redis pods bootstrap correctly (one master, two replicas).
*   Test failover by killing the master pod and observing Sentinel logs and new master election.
*   Update the `RAMPOS_REDIS_URL` environment variable in deployment if using a Sentinel-aware connection string (e.g. `redis+sentinel://:password@sentinel-host:26379/mymaster`).

## Notes
*   The `redis` crate in Rust has varying levels of support for Sentinel. The current implementation assumes the connection string passed to `Client::open` is sufficient, or that a separate HA proxy (like Envoy or HAProxy) sits in front if the client doesn't support it natively. For strict client-side Sentinel support, a more complex client builder might be needed in the future, but `redis-rs` supports `redis+sentinel://` scheme in newer versions.
