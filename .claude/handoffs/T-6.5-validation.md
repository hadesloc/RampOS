# Handoff: T-6.5 Request Validation Middleware

## Summary

Implemented comprehensive request validation middleware for the RampOS API, ensuring all incoming request bodies are validated before processing with detailed, structured error responses.

## Changes Made

### 1. Enhanced ValidatedJson Extractor (`crates/ramp-api/src/extract.rs`)

**Before:** Simple validation that returned generic error messages.

**After:**
- Structured validation error response with field-level details
- Custom `ValidationRejection` type with formatted error details
- Separate handling for JSON parsing errors vs validation errors
- Nested field support (e.g., `bank_account.account_number`)
- Human-readable default messages for common validation rules (length, range, email, url, regex)

**Error Response Format:**
```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Validation failed for 2 fields",
    "details": {
      "tenant_id": [{"code": "length", "message": "Tenant ID must be 1-64 characters"}],
      "amount_vnd": [{"code": "range", "message": "Value must be at least 1000"}]
    }
  }
}
```

### 2. Enhanced DTOs with Validation Rules (`crates/ramp-api/src/dto.rs`)

Added comprehensive validation to all request DTOs:

**Custom Validators:**
- `validate_alphanumeric_underscore` - For tenant_id, user_id fields
- `validate_bank_account_number` - Digits only, 6-20 characters
- `validate_trading_symbol` - Format like BTC/VND
- `validate_metadata` - JSON size limit (64KB)
- `validate_vn_bank_code` - Vietnamese bank code format
- `validate_eth_address` - Ethereum address format (0x + 40 hex chars)
- `validate_hex_string` - Hex strings with 0x prefix

**Validated DTOs:**
- `CreatePayinRequest` - amount range 1000-500000000 VND
- `ConfirmPayinRequest` - all fields validated
- `CreatePayoutRequest` - amount range 10000-500000000 VND, nested bank_account validation
- `TradeExecutedRequest` - trading symbol validation
- `BankAccountDto` - bank code and account number format
- `CreateTenantRequest` - name and config validation
- `UpdateTenantRequest` - URL validation for webhook
- `SuspendTenantRequest` - reason validation
- `TierChangeRequest` - tier and reason validation
- Account Abstraction DTOs - Ethereum address and hex string validation

**OpenAPI Schema Annotations:**
- Added `#[schema(example = ...)]` with realistic examples
- Added `min_length`, `max_length`, `minimum`, `maximum` constraints
- Added documentation comments for all fields

## Validation Rules Summary

| Field Type | Validation |
|------------|------------|
| tenant_id, user_id | 1-64 chars, alphanumeric + underscore + hyphen |
| amount_vnd (payin) | 1,000 - 500,000,000 VND |
| amount_vnd (payout) | 10,000 - 500,000,000 VND |
| rails_provider | 1-32 chars |
| bank_code | 2-32 chars, alphanumeric |
| account_number | 6-20 digits only |
| account_name | 1-255 chars |
| trading symbol | Format: BASE/QUOTE (e.g., BTC/VND) |
| metadata | Max 64KB JSON |
| webhook_url | Valid URL, max 2048 chars |
| Ethereum addresses | 42 chars, 0x prefix + 40 hex chars |
