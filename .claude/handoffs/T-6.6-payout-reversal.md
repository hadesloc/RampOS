# Task T-6.6: Payout Reversal Logic - Handoff Report

## Summary

Implemented complete payout reversal logic for handling bank rejections, timeouts, and partial success scenarios. Funds are properly returned to users when payouts fail.

## Changes Made

### 1. PayoutState Enum (ramp-common/src/intent.rs)

Added `Reversed` state to track when funds have been returned to users:

```rust
pub enum PayoutState {
    // ... existing states ...
    // Reversal state - funds returned to user
    Reversed,
}
```

Updated state machine:
- `BankRejected` can now transition to `Reversed`
- `Timeout` can now transition to `Reversed`
- `Cancelled` can now transition to `Reversed`
- `ManualReview` can now transition to `Reversed`
- `Reversed` is a terminal state

Added `requires_reversal()` helper method.

### 2. Ledger Patterns (ramp-common/src/ledger.rs)

Added two new ledger transaction patterns:

**`payout_vnd_reversed`** - Full reversal:
- Debits `ClearingBankPending` (release held funds)
- Credits `LiabilityUserVnd` (refund to user)

**`payout_vnd_partial_reversed`** - Partial reversal:
- For partial settlement scenarios
- Calculates difference between original and settled amount
- Returns only the unsettled portion to user

### 3. PayoutWorkflow (ramp-core/src/workflows/payout.rs)

**New types:**
- `ReversalReason` enum with variants:
  - `BankRejected(String)`
  - `Timeout`
  - `PartialSuccess { settled_amount: Decimal }`
  - `SubmissionFailed(String)`
  - `Cancelled`

**Updated `SettlementResult`:**
- Added `settled_amount: Option<i64>` for partial success scenarios

**Enhanced `reverse_funds` method:**
- Uses proper ledger patterns instead of manual entry creation
- Handles both full and partial reversals
- Logs reversal operations for audit trail

**Updated workflow handling:**
- Bank rejection: Reverses funds, updates state to `REVERSED`, publishes webhook
- Timeout: Reverses funds instead of leaving in limbo
- Partial success: Finalizes settled amount, reverses remainder
- Submission failure: Reverses immediately

### 4. PayoutService (ramp-core/src/service/payout.rs)

Updated `confirm_payout` for bank rejection case:
- Creates reversal ledger transaction
- Updates state to `REVERSED`
- Publishes `payout.reversed` webhook event

Updated `parse_payout_state` to include `REVERSED`.

### 5. EventPublisher (ramp-core/src/event.rs)

Added `publish_payout_reversed` method to:
- `EventPublisher` trait
- `NatsEventPublisher` implementation
- `InMemoryEventPublisher` implementation

### 6. Test Utilities (ramp-core/src/test_utils.rs)

Updated `list_expired` to include `REVERSED` in terminal states.

## Ledger Entry Flow

### Normal Payout (Success)
```
1. payout_vnd_initiated:
   DR LiabilityUserVnd (user)   1,000,000
   CR ClearingBankPending       1,000,000

2. payout_vnd_confirmed:
   DR ClearingBankPending       1,000,000
   CR AssetBank                 1,000,000
```

### Payout Reversal (Bank Rejected)
```
1. payout_vnd_initiated:
   DR LiabilityUserVnd (user)   1,000,000
   CR ClearingBankPending       1,000,000

2. payout_vnd_reversed:
   DR ClearingBankPending       1,000,000
   CR LiabilityUserVnd (user)   1,000,000
```

### Partial Payout (700,000 settled, 300,000 returned)
```
1. payout_vnd_initiated:
   DR LiabilityUserVnd (user)   1,000,000
   CR ClearingBankPending       1,000,000

2. payout_vnd_confirmed (partial):
   DR ClearingBankPending         700,000
   CR AssetBank                   700,000

3. payout_vnd_partial_reversed:
   DR ClearingBankPending         300,000
   CR LiabilityUserVnd (user)     300,000
```
