## Deployment Pipeline Features

### 1. GitHub Actions Workflow (`deploy.yaml`)

#### Triggers
- **Push to `develop`**: Auto-deploys to dev environment
- **Push to `main`**: Auto-deploys to staging environment
- **Push version tag `v*`**: Triggers production deployment (with approval)
- **Manual dispatch**: Select environment (dev/staging/production) and optional image tag

#### Jobs

| Job | Environment | Trigger | Approval |
|-----|-------------|---------|----------|
| `build` | - | All triggers | No |
| `deploy-dev` | dev | develop branch | No |
| `deploy-staging` | staging | main branch | No |
| `deploy-production` | production | v* tags or manual | **Required** |
| `rollback` | Any | Manual | Per-env |

#### Key Features
- **Docker Buildx** with GitHub Actions cache (GHA)
- **SBOM and provenance** generation for supply chain security
- **Image metadata** with semantic versioning and SHA tags
- **Pre-deployment backup** annotations for production
- **Rollback tracking** (previous image captured before deploy)
- **Health checks** with retries
- **GitHub Release notes** auto-generation for version tags

### 2. Environment Promotion Flow

```
develop branch   -->  [Build]  -->  [Deploy Dev]
                              |
main branch      -->  [Build]  -->  [Deploy Staging]
                              |
v* tag           -->  [Build]  -->  [Deploy Staging]  -->  [Manual Approval]  -->  [Deploy Prod]
```

### 3. ArgoCD Configuration

#### Applications
| Application | Branch | Namespace | Auto-Sync |
|-------------|--------|-----------|-----------|
| `rampos-dev` | develop | rampos-dev | Yes (prune) |
| `rampos-staging` | main | rampos-staging | Yes (prune) |
| `rampos` (prod) | HEAD | rampos | Yes (self-heal only) |

#### Project RBAC
- **developer**: Get all apps, sync dev/staging
- **deployer**: Full access to all environments
- **admin**: Full control including project settings

### 4. Rollback Capability

#### Via GitHub Actions
```bash
# Rollback production to specific image
gh workflow run deploy.yaml \
  -f environment=production \
  -f image_tag=sha-abc123...

# Rollback staging
gh workflow run deploy.yaml \
  -f environment=staging \
  -f image_tag=sha-abc123...
```

#### Via ArgoCD
```bash
# Sync to previous revision
argocd app rollback rampos <revision>

# Or using history
argocd app history rampos
argocd app rollback rampos 5  # rollback to revision 5
```

#### Via kubectl
```bash
# Using rollout history
kubectl rollout history deployment/rampos-server -n rampos
kubectl rollout undo deployment/rampos-server -n rampos

# Or to specific revision
kubectl rollout undo deployment/rampos-server -n rampos --to-revision=3
```

## Required Secrets

Configure these in GitHub repository settings (Settings > Secrets > Actions):

| Secret | Environment | Description |
|--------|-------------|-------------|
| `KUBECONFIG_DEV` | dev | Base64-encoded kubeconfig for dev cluster |
| `KUBECONFIG_STAGING` | staging | Base64-encoded kubeconfig for staging cluster |
| `KUBECONFIG_PROD` | production | Base64-encoded kubeconfig for prod cluster |
| `ARGOCD_SERVER` | production | ArgoCD server URL (optional) |
| `ARGOCD_AUTH_TOKEN` | production | ArgoCD API token (optional) |

### Generate kubeconfig secret
```bash
cat ~/.kube/config | base64 -w 0
```

## GitHub Environments Setup

Create these environments in GitHub (Settings > Environments):

1. **dev**
   - No protection rules
   - Add `KUBECONFIG_DEV` secret

2. **staging**
   - No protection rules (or optional reviewers)
   - Add `KUBECONFIG_STAGING` secret

3. **production**
   - **Required reviewers**: Add approvers
   - **Wait timer**: Optional (e.g., 10 minutes)
   - **Deployment branches**: Only `main` and `v*` tags
   - Add `KUBECONFIG_PROD`, `ARGOCD_SERVER`, `ARGOCD_AUTH_TOKEN` secrets
