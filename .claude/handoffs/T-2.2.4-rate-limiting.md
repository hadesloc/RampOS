# Handoff: Rate Limiting Middleware (T-2.2.4)

## Summary
Implemented a sliding window rate limiting middleware using Redis. The middleware enforces both global and per-tenant limits, with support for per-endpoint overrides.

## Changes
- Created `crates/ramp-api/src/middleware/rate_limit.rs`
- Updated `crates/ramp-api/src/middleware/mod.rs`
- Updated `crates/ramp-api/src/router.rs` to include the middleware in the pipeline
- Updated `crates/ramp-api/src/main.rs` to initialize Redis connection and RateLimiter

## Configuration
Default limits:
- Global: 1000 requests/minute
- Per-tenant: 100 requests/minute
- Window: 60 seconds

Configuration can be adjusted in `RateLimitConfig`.

## Testing
- Unit tests added in `crates/ramp-api/src/middleware/rate_limit.rs` and `crates/ramp-api/src/middleware/rate_limit_test.rs`
- Verified logic via unit tests. Integration tests require a running Redis instance.

## Usage
The middleware is automatically applied to all `/v1` routes if Redis is available. It checks limits in the following order:
1. Global limit
2. Tenant limit
3. Endpoint-specific limit (if configured)

Rate limit headers are added to the response:
- `X-RateLimit-Limit`
- `X-RateLimit-Remaining`
- `X-RateLimit-Reset`
