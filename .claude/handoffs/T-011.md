## Invariants Coverage

The invariant tests cover the following critical security properties:

### Balance Invariants
- Account balance is always non-negative (implicit via EVM)
- Value transferred never exceeds funded amount
- Accounts can always receive ETH

### Access Control Invariants
- Owner address is never zero and remains constant
- Entry point reference is immutable
- Verifying signer is never zero
- Only owner can modify session keys (enforced via handler)

### State Machine Invariants
- Session key count tracking is consistent
- Removed session keys are invalid
- Withdrawal timelock is enforced (24 hours)
- Withdrawal state transitions are valid
- Pending withdrawal amount and recipient are consistent

### Factory Invariants
- Implementation is immutable
- Address prediction is deterministic
- Different inputs produce different addresses
- All accounts have valid code and owners

## Notes

1. The tests use handler contracts that expose bounded actions for fuzzing, ensuring valid state transitions.

2. Ghost variables track cumulative state changes for cross-invariant verification.

3. Pre-existing unit test failures in `RampOSPaymaster.t.sol` (test_TenantLimit, test_ValidateUserOp) are unrelated to the invariant tests - they appear to be caused by signature validation changes from previous security fixes.

4. The invariant tests found an edge case where `requestWithdraw(0)` sets a recipient but with zero amount - this was documented as expected behavior per the contract implementation.
