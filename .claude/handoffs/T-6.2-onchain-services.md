**Structs:**

- `CreateDepositResponse` - Response with intent ID, deposit address, status
- `ConfirmDepositRequest` - Request to update confirmations
- `KytCheckRequest` - KYT (Know Your Transaction) check request
- `DepositService` - Main service struct

**Methods:**

- `update_confirmations()` - Update confirmation count, transition to CONFIRMED when sufficient
- `process_kyt_check()` - Run KYT check, flag high-risk deposits
- `credit_deposit()` - Credit crypto to user's ledger balance
- `complete_deposit()` - Mark deposit as completed
- `process_confirmed_deposit()` - Full automated flow: KYT -> Credit -> Complete

**State Flow:**
```
DETECTED -> CONFIRMING -> CONFIRMED -> KYT_CHECKED -> CREDITED -> COMPLETED
                              |
                              v
                         KYT_FLAGGED -> MANUAL_REVIEW -> (CREDITED | REJECTED)
```

**Features:**
- Idempotency support via idempotency_key
- Chain-specific confirmation requirements (Ethereum: 12, Polygon: 128, L2s: 1)
- Configurable KYT threshold (default 0.7)
- Event publishing for status changes
- Full test coverage (7 tests)

### 3. `crates/ramp-core/src/service/withdraw.rs` (NEW)

Complete WithdrawService implementation:

**Structs:**
- `CreateWithdrawRequest` - Request to create withdraw intent
- `CreateWithdrawResponse` - Response with intent ID, status, estimated gas
- `ExecuteWithdrawRequest` - Request to execute UserOp submission
- `ConfirmWithdrawRequest` - Request to confirm when tx mined
- `WithdrawKytRequest` - KYT check for destination address
- `WithdrawService` - Main service struct

**Methods:**
- `create_withdraw()` - Create withdraw intent, check balance, hold funds
- `process_kyt_check()` - KYT check for destination address
- `mark_signed()` - Mark UserOp as signed
- `execute_withdraw()` - Submit UserOp to bundler, update to BROADCASTED
- `confirm_withdraw()` - Confirm when tx mined, finalize or reverse ledger
- `cancel_withdraw()` - Cancel and return held funds

**State Flow:**
```
CREATED -> POLICY_APPROVED -> KYT_CHECKED -> SIGNED -> BROADCASTED -> CONFIRMING -> CONFIRMED -> COMPLETED
   |              |                |           |
   v              v                v           v
REJECTED     KYT_FLAGGED    CANCELLED   BROADCAST_FAILED (funds returned)
```

**Features:**
- Idempotency support
- Balance validation before creating intent
- Address validation (EVM format)
- Maximum withdraw amount limit (configurable)
- Automatic fund hold on POLICY_APPROVED (ledger entry)
- Automatic fund return on cancellation or failure
- Configurable KYT threshold
- Event publishing
- Full test coverage (9 tests)

## Ledger Double-Entry Patterns

All ledger patterns maintain double-entry balance:

### Deposit Confirmed
```
DEBIT  Asset:Crypto              [amount]  <- We now hold the crypto
CREDIT Liability:UserCrypto      [amount]  <- We owe user the crypto
```

### Withdraw Initiated (Hold)
```
DEBIT  Liability:UserCrypto      [amount]  <- Reduce user's balance
CREDIT Clearing:CryptoPending    [amount]  <- Hold in clearing
```

### Withdraw Confirmed
```
DEBIT  Clearing:CryptoPending    [amount]  <- Clear the hold
CREDIT Asset:Crypto              [amount]  <- Crypto left our custody
```

### Withdraw Reversed (Failed/Cancelled)
```
DEBIT  Clearing:CryptoPending    [amount]  <- Clear the hold
CREDIT Liability:UserCrypto      [amount]  <- Return to user's balance
```
