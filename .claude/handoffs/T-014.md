## Failover Procedures

### Automatic Failover
The HA setup includes health checks that trigger automatic failover:
1. Startup probe allows 5 minutes for replica catch-up
2. Liveness probe checks every 10s (6 failures = restart)
3. Readiness probe removes unhealthy pods from service

### Manual Failover (Emergency)
```bash
# When primary is down and won't recover
kubectl exec -n rampos rampos-postgres-1 -- touch /tmp/postgresql.trigger

# Update labels
kubectl label pod -n rampos rampos-postgres-1 role=primary --overwrite
kubectl label pod -n rampos rampos-postgres-0 role=replica --overwrite
```

### Planned Switchover (Graceful)
```bash
# Use the switchover script
kubectl exec -n rampos <any-pod> -- /scripts/switchover.sh rampos-postgres-0 rampos-postgres-1
```

### Post-Failover Steps
1. Verify new primary is accepting writes
2. Reconfigure old primary as replica:
   - Create recovery.signal file
   - Update primary_conninfo in postgresql.auto.conf
   - Restart PostgreSQL
3. Verify replication is working
4. Update monitoring dashboards
